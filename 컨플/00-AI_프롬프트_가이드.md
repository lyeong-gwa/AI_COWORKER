## 1부: 기초 개념 설명

### LLM(대형 언어 모델)이란?

**한 줄 정의**: 사람처럼 글을 읽고 쓸 수 있도록 학습된 거대한 AI 시스템입니다.

ChatGPT, Claude, GPT-4 등이 모두 LLM입니다. 이들은 엄청난 양의 텍스트로 학습되어, 질문에 답하고, 문서를 요약하고, 코드를 작성하는 등 다양한 언어 작업을 수행할 수 있습니다.

### 프롬프트(Prompt)란?

**비유**: AI에게 보내는 "지시서" 또는 "업무 요청서"

프롬프트는 LLM에게 전달하는 입력 텍스트입니다. 마치 직원에게 업무를 지시하듯, AI에게 무엇을 해야 할지 설명하는 글입니다.

**예시**:
```
프롬프트: "다음 이메일을 정중하게 답변해주세요: ..."
→ LLM이 답변 생성
```

좋은 프롬프트는 명확하고 구체적이며, 원하는 결과를 정확히 얻을 수 있게 합니다.

### RAG(Retrieval-Augmented Generation)란?

**비유**: "오픈북 시험" - AI가 시험 볼 때 교과서를 펼쳐보고 답하는 것

RAG는 **검색(Retrieval) + 생성(Generation)**의 결합입니다:

1. **검색 단계**: 질문과 관련된 문서를 찾아옴 (교과서에서 관련 페이지 찾기)
2. **생성 단계**: 찾은 문서를 참고하여 답변 작성 (교과서를 보고 답안 작성)

**왜 필요한가?**
- LLM은 학습 당시의 지식만 알고 있음 (2023년 이전 데이터)
- 우리 회사의 내부 문서나 최신 정보는 모름
- RAG로 실시간 문서를 주입하면, AI가 **최신 정보와 회사 지식을 바탕으로 답변** 가능

### 벡터 임베딩(Vector Embedding)이란?

**비유**: "문서의 지문" - 텍스트를 숫자 배열로 변환한 것

컴퓨터는 텍스트를 직접 비교할 수 없습니다. 그래서 텍스트를 숫자 배열(벡터)로 변환합니다.

**예시**:
```
"파이썬 프로그래밍" → [0.2, 0.8, 0.1, ..., 0.5] (384개 숫자)
"자바 개발"        → [0.3, 0.7, 0.2, ..., 0.4]
"점심 메뉴"        → [0.9, 0.1, 0.3, ..., 0.8]
```

비슷한 내용은 비슷한 숫자 패턴을 갖게 됩니다. "파이썬"과 "자바"는 숫자가 비슷하지만, "점심 메뉴"는 전혀 다릅니다.

**우리 시스템 정보**:
- 임베딩 모델: `sentence-transformers/all-MiniLM-L6-v2`
- 벡터 차원: 384차원 (384개의 숫자로 표현)
- 실행 방식: ONNX 로컬 실행 (외부 서버 필요 없음)

### 유사도 검색이란?

**비유**: "도서관 사서가 관련 책 찾아주는 것"

벡터 임베딩을 이용해 비슷한 내용의 문서를 찾는 기술입니다.

**작동 방식**:
1. 사용자 질문을 벡터로 변환: `[0.2, 0.8, 0.1, ...]`
2. 저장된 모든 문서 벡터와 거리 계산 (코사인 유사도)
3. 가장 가까운 문서 3개 선택
4. 유사도 점수와 함께 반환

**우리 시스템 설정**:
```python
top_k = 3           # 상위 3개 문서 반환
min_score = 0.3     # 최소 유사도 30% 이상
distance = "cosine" # 코사인 유사도 사용
```

**유사도 점수 해석**:
- `0.85~1.0`: 매우 관련 높음 (거의 같은 주제)
- `0.6~0.85`: 관련 있음 (참고 가능)
- `0.3~0.6`: 약간 관련 (추가 맥락)
- `0~0.3`: 거의 무관 (제외됨)

## 2부: 우리 시스템의 프롬프트 관리 체계

### 프롬프트 3종 세트

우리 시스템은 **3가지 프롬프트**를 사용합니다:

| 프롬프트 종류 | 역할 | 파일 위치 | Temperature | Max Tokens |
|--------------|------|-----------|-------------|------------|
| **의도 감지 프롬프트** | 사용자가 무엇을 원하는지 파악 | `agent_service.py` 라인 253 | 0.1 (정확성) | 500 |
| **어시스턴트 응답 프롬프트** | 사용자에게 답변 생성 | `agent_service.py` 라인 636 | 0.7 (창의성) | 1000 |
| **댓글 자동 생성 프롬프트** | 태스크 댓글 작성 | `agent_service.py` 라인 222 | 0.5 (중간) | 800 |

### 1. 의도 감지 프롬프트 (INTENT_DETECTION_PROMPT)

**위치**: `backend/app/services/agent_service.py` 라인 253-292

**역할**: 사용자 메시지를 분석하여 무엇을 원하는지 파악

**파라미터**:
- `temperature=0.1`: 가장 낮음 (정확한 분류 필요)
- `max_tokens=500`: 짧은 JSON 응답만 필요

**감지하는 액션 타입**:
- `explain`: 설명 요청 ("이게 뭐야?")
- `view`: 조회 요청 ("보여줘")
- `create`: 생성 요청 ("태스크 만들어줘")
- `update`: 수정 요청 ("상태 변경해줘")
- `delete`: 삭제 요청
- `execute`: 실행 요청
- `search`: 검색 요청 ("코드아이즈 점검 방법 알려줘")
- `chat`: 일반 대화

**감지하는 타겟 타입**:
- `task`: 태스크
- `tool`: 도구
- `node`: 노드
- `workflow`: 워크플로우
- `document`: 문서

**출력 형식** (JSON):
```json
{
  "action": "create",
  "target": "task",
  "parameters": {
    "title": "민원: 점검결과 이상",
    "description": "...",
    "priority": "high",
    "tags": ["민원", "코드아이즈"],
    "addComment": true,
    "searchCategory": "코드아이즈"
  },
  "confidence": 0.9
}
```

### 2. 어시스턴트 응답 프롬프트 (ASSISTANT_SYSTEM_PROMPT)

**위치**: `backend/app/services/agent_service.py` 라인 636-654

**역할**: 사용자에게 친절하고 유용한 답변 제공

**파라미터**:
- `temperature=0.7`: 중간 (자연스러운 대화체)
- `max_tokens=1000`: 충분한 설명 제공

**시스템 프롬프트 내용**:
```
당신은 'AI 업무도우미'의 AI 어시스턴트입니다.
사용자의 업무 관리를 돕는 친절하고 유능한 비서 역할을 합니다.

주요 기능:
- 태스크(Task) 관리
- 도구(Tool) 설명 및 활용
- AI 노드(Node) 설정
- 워크플로우(Workflow) 관리
- 지식 문서(Document) 검색

응답 가이드:
- 한국어로 답변
- 간결하고 명확하게
- 작업 완료 시 결과 확인
- 친절하지만 과하지 않은 톤
- 지식 베이스 정보 근거로 정확한 답변
```

### 3. 댓글 자동 생성 프롬프트

**위치**: `backend/app/services/agent_service.py` 라인 222-229

**역할**: 태스크에 대한 자동 답변(댓글) 작성

**파라미터**:
- `temperature=0.5`: 중간 (정확성과 자연스러움 균형)
- `max_tokens=800`: 충분한 답변 길이

**시스템 프롬프트 내용**:
```
당신은 업무 민원에 대한 답변을 작성하는 전문가입니다.
주어진 태스크 내용과 관련 지식을 참고하여
정중하고 전문적인 답변 초안을 작성하세요.

규칙:
- 관련 지식이 있으면 반드시 근거로 활용
- 구체적이고 실행 가능한 안내 제공
- 정중하지만 간결한 톤
- 확인된 사실만 언급, 추측은 "확인이 필요합니다"로 표현
```

### 프롬프트 수정 방법

**1단계**: 파일 열기
```bash
backend/app/services/agent_service.py
```

**2단계**: 해당 라인 찾기
- 의도 감지: 라인 253
- 어시스턴트 응답: 라인 636
- 댓글 생성: 라인 222

**3단계**: 프롬프트 텍스트 수정

**4단계**: 서버 재시작
```bash
cd backend
python -m uvicorn app.main:app --reload
```

**주의사항**:
- Temperature 값은 함께 고려해야 함 (프롬프트만 바꿔도 효과는 있지만)
- JSON 출력 형식을 바꾸면 파싱 로직도 수정 필요
- 테스트 후 배포 권장

## 3부: 사용자 메시지가 AI 응답이 되기까지

### 전체 흐름도

```
[사용자]
   |
   | "코드아이즈 보안 취약점 점검 방법 알려줘"
   v
[프론트엔드: ChatAssistant 컴포넌트]
   |
   | POST /api/chat/messages
   | {content: "...", context: {type: "none"}, sessionId: "..."}
   v
[백엔드: /api/chat/messages 엔드포인트]
   |
   | agent_service.process_chat_message()
   v
[1단계: 세션 관리]
   | - 세션 ID 생성/조회
   | - 사용자 메시지를 세션에 저장
   v
[2단계: 컨텍스트 로딩]
   | - context.type이 'task', 'document' 등이면 DB에서 조회
   | - 없으면 None
   v
[3단계: 의도 감지]
   |
   | LLM 호출 #1 (의도 감지)
   | ┌─────────────────────────────────────┐
   | │ System: INTENT_DETECTION_PROMPT     │
   | │ User: "코드아이즈 보안 취약점..."    │
   | │       + 컨텍스트 정보 (있다면)       │
   | └─────────────────────────────────────┘
   |
   | 응답: {action: "search", target: "document",
   |        parameters: {searchCategory: "코드아이즈"}}
   v
[3.5단계: RAG 검색]
   |
   | search_knowledge_base()
   | ┌─────────────────────────────────────┐
   | │ 1. 쿼리 벡터화                       │
   | │    "코드아이즈 보안 취약점..."       │
   | │    → [0.2, 0.8, ...] (384차원)      │
   | │                                      │
   | │ 2. ChromaDB 코사인 유사도 검색       │
   | │    - top_k=3                         │
   | │    - min_score=0.3                   │
   | │    - where={category: "코드아이즈"}  │
   | │                                      │
   | │ 3. 결과 반환                         │
   | │    [                                 │
   | │      {id: "doc-1", score: 0.87, ... }│
   | │      {id: "doc-2", score: 0.72, ... }│
   | │      {id: "doc-3", score: 0.65, ... }│
   | │    ]                                 │
   | └─────────────────────────────────────┘
   |
   | format_knowledge_context()
   | → "[관련 지식]\n1. (유사도: 0.87) 코드아이즈 취약점 점검 가이드\n내용..."
   v
[4단계: 작업 실행]
   |
   | execute_action()
   | - action="search"이면 검색 결과 정리
   | - action="create"이면 DB에 태스크 생성
   | - action="update"이면 DB 업데이트
   |
   | 결과: AgentAction {type: "search", success: true, result: {...}}
   v
[5단계: 응답 생성]
   |
   | LLM 호출 #2 (응답 생성)
   | ┌─────────────────────────────────────┐
   | │ messages = [                         │
   | │   {role: "system",                   │
   | │    content: ASSISTANT_SYSTEM_PROMPT},│
   | │                                      │
   | │   {role: "system",                   │
   | │    content: "[관련 지식]\n1. ..."},  │
   | │                                      │
   | │   ...세션 히스토리 (최근 10개)...     │
   | │                                      │
   | │   {role: "user",                     │
   | │    content: "코드아이즈 보안..."},   │
   | │                                      │
   | │   {role: "system",                   │
   | │    content: "[시스템] 작업 완료..."} │
   | │ ]                                    │
   | └─────────────────────────────────────┘
   |
   | 응답: "코드아이즈 보안 취약점 점검은 다음과 같이 진행됩니다..."
   v
[6단계: 응답 반환]
   |
   | 응답을 세션에 저장
   | ChatMessageResponse 생성
   |
   | {
   |   id: "msg-abc123",
   |   role: "assistant",
   |   content: "코드아이즈 보안 취약점 점검은...",
   |   action: {type: "search", success: true, ...},
   |   sessionId: "session-xyz"
   | }
   v
[프론트엔드]
   |
   | 메시지 화면에 표시
   | 마크다운 렌더링 (react-markdown)
   v
[사용자에게 표시]
```

### 각 단계 상세 설명

#### 1단계: 세션 관리
- 세션 ID가 없으면 생성 (`session-{랜덤12자리}`)
- 세션 ID로 메시지 히스토리 조회
- 사용자 메시지를 히스토리에 추가

#### 2단계: 컨텍스트 로딩
- 사용자가 특정 항목(태스크, 문서 등)을 선택했으면 DB에서 조회
- 예: task-123 선택 → Task 테이블에서 제목, 설명, 상태 등 가져옴

#### 3단계: 의도 감지
- LLM에게 사용자 메시지 + 컨텍스트 정보 전달
- "무엇을 원하는가?" 분석
- JSON 형식으로 action, target, parameters 반환

#### 3.5단계: RAG 검색
- 질문 내용으로 벡터 DB 검색
- 카테고리 필터 적용 (searchCategory가 있으면)
- 상위 3개 문서 선택 (유사도 0.3 이상)
- 포맷: `[관련 지식]\n1. (유사도: 0.87) 제목\n내용...`

#### 4단계: 작업 실행
- 의도에 따라 실질적인 작업 수행
- DB에 데이터 생성/수정/삭제
- 검색 결과 정리
- 성공/실패 여부 기록

#### 5단계: 응답 생성
- 모든 정보를 종합하여 LLM에 전달
- 지식 베이스 검색 결과 포함
- 세션 히스토리 포함 (최근 10개)
- 작업 결과 포함
- 자연스러운 한국어 답변 생성

#### 6단계: 응답 반환
- 생성된 답변을 세션에 저장
- 프론트엔드에 JSON 응답 전송
- 화면에 렌더링

## 4부: LLM이 최종적으로 받는 프롬프트 예시

### 시나리오: "코드아이즈 보안 취약점 점검 방법 알려줘"

#### 1단계: 의도 감지 LLM 호출

**LLM에 전달되는 메시지**:

```python
messages = [
    {
        "role": "system",
        "content": """당신은 사용자의 의도를 파악하는 AI입니다.
사용자 메시지와 선택된 컨텍스트를 분석하여 의도를 JSON으로 반환하세요.

가능한 action 타입:
```

## 5부: 워크플로우 노드의 프롬프트

### 워크플로우 vs 채팅의 차이

| 구분 | 채팅 어시스턴트 | 워크플로우 노드 |
|------|----------------|----------------|
| **목적** | 사용자와 대화 | 자동화된 작업 수행 |
| **프롬프트** | 시스템이 자동 구성 | 사용자가 직접 작성 |
| **지식 검색** | 자동 RAG (유사도 검색) | ID 기반 직접 조회 |
| **입력** | 사용자 메시지 | 이전 노드 출력 |
| **출력** | 자연어 답변 | JSON/텍스트 (다음 노드로) |

### 워크플로우 노드 프롬프트 구조

**위치**: `backend/app/services/node_executor.py` 라인 243-286

워크플로우 노드는 **템플릿 변수**를 사용합니다:

```
{{input.xxx}}         # 입력 데이터
{{toolResults.xxx}}   # 도구 실행 결과
{{knowledge}}         # 연결된 지식 베이스 내용
```

### 템플릿 변수 설명

#### 1. `{{input.fieldName}}`

**예시**:

```json

  "content": "로그인이 안 됩니다"
}
```

```
{{input.content}}
```

```
다음 문의에 대한 답변을 작성하세요:
```

## 6부: 우리 시스템의 LLM 지원 현황

### 지원하는 LLM 프로바이더 (4종)

| 프로바이더 | 환경변수 | 추천 모델 | 용도 |
|-----------|---------|----------|------|
| **OpenAI** | `OPENAI_API_KEY` | gpt-4o-mini, gpt-4o, gpt-4-turbo | 기본 추천 |
| **Azure OpenAI** | `AZURE_OPENAI_ENDPOINT`, `AZURE_OPENAI_API_KEY` | gpt-4o, gpt-4-turbo | 엔터프라이즈 |
| **Anthropic** | `ANTHROPIC_API_KEY` | claude-3-5-sonnet, claude-3-opus | 고품질 답변 |
| **Custom API** | `CUSTOM_API_URL`, `CUSTOM_API_KEY` | (자체 서버) | 온프레미스 |

### 프로바이더 자동 선택 우선순위

**위치**: `backend/app/services/llm/registry.py` 라인 86-112

시스템은 다음 순서로 사용 가능한 프로바이더를 자동 선택합니다:

```
1. CUSTOM_API_URL + CUSTOM_API_KEY 설정됨
   → custom_api 사용

2. AZURE_OPENAI_ENDPOINT + AZURE_OPENAI_API_KEY 설정됨
   → azure 사용

3. OPENAI_API_KEY 설정됨
   → openai 사용 (기본값)

4. ANTHROPIC_API_KEY 설정됨
   → anthropic 사용

5. 아무것도 없음
   → 오류 발생 (환경변수 설정 필요)
```

## 부록: 용어 사전

### A-L

**AI (Artificial Intelligence, 인공지능)**
인간의 학습, 추론, 지각 능력을 컴퓨터로 구현한 기술

**API (Application Programming Interface)**
소프트웨어 간 통신 규약. LLM API는 HTTP 요청으로 AI와 통신

**Async/Await (비동기 처리)**
작업 완료를 기다리지 않고 다른 작업을 동시에 수행하는 프로그래밍 방식

**ChromaDB**
벡터 데이터베이스. 임베딩 벡터를 저장하고 유사도 검색 수행

**Context (컨텍스트)**
대화나 작업의 배경 정보. 선택된 태스크, 이전 대화 내역 등

**Cosine Similarity (코사인 유사도)**
두 벡터 간 각도로 유사도를 측정. 1에 가까울수록 유사

**Embedding (임베딩)**
텍스트를 고차원 벡터(숫자 배열)로 변환하는 과정

**Endpoint (엔드포인트)**
API 서버의 특정 기능에 접근하는 URL 경로 (예: `/api/chat/messages`)

**JSON (JavaScript Object Notation)**
데이터 교환 형식. `{"key": "value"}` 구조

**LLM (Large Language Model, 대형 언어 모델)**
대규모 텍스트로 학습된 AI 모델. GPT, Claude 등

### M-Z

**Max Tokens (최대 토큰)**
LLM이 생성할 수 있는 최대 텍스트 길이 제한

**Message (메시지)**
LLM과 주고받는 대화 단위. role(역할) + content(내용)

**Metadata (메타데이터)**
데이터에 대한 부가 정보. 문서의 카테고리, 태그, 작성일 등

**ONNX (Open Neural Network Exchange)**
AI 모델 실행 형식. 오프라인으로 로컬에서 빠르게 실행 가능

**Prompt (프롬프트)**
LLM에게 보내는 지시문. 질문, 명령, 배경 정보 포함

**RAG (Retrieval-Augmented Generation)**
검색 기반 생성. 관련 문서를 찾아 참고하여 답변 생성

**Schema (스키마)**
데이터 구조 정의. JSON Schema로 입출력 형식 지정

**Session (세션)**
사용자의 대화 기록을 추적하는 단위. 연속된 대화를 기억

**System Prompt (시스템 프롬프트)**
LLM의 역할과 행동 방식을 정의하는 최상위 지시문

**Temperature (온도)**
LLM의 창의성 조절 파라미터. 0=결정적, 2=매우 창의적

**Token (토큰)**
텍스트 처리 단위. 영어는 1단어≈1토큰, 한글은 1글자≈1~2토큰

**Top-K**
검색 시 반환할 상위 결과 개수 (예: top_k=3 → 상위 3개)

**Vector (벡터)**
숫자 배열. 임베딩으로 변환된 텍스트의 수치 표현

**Vector Database (벡터 데이터베이스)**
벡터를 저장하고 유사도 검색을 수행하는 특수 데이터베이스

**Workflow (워크플로우)**
여러 단계의 작업을 자동화한 프로세스. 노드들이 순차/병렬 실행
